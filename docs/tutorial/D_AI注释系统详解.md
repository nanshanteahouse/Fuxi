# Fuxi AI 注释系统详解

> 细胞类型注释是单细胞分析中最核心、最耗时的步骤之一。Fuxi 的 AI 注释引擎让你从"逐个聚类查阅文献"转向"AI 给出专业初稿、专家审核修订"的工作模式。

## 为什么需要 AI 注释？

在[RNA 流程 Step 05](09_细胞类型注释.md) 中，我们介绍了三种注释模式：

1. **Knowledge Base 模式**：使用手工整理的标记基因集进行打分和专家规则匹配
2. **AI 模式**：调用 LLM 进行注释
3. **Score_genes 模式**：简单的 `sc.tl.score_genes()` 回调

KB 模式是最可靠、最可解释的方式——但它有严重的局限性：

- **构建 KB 需要专家投入大量时间。** 一个完整的视网膜 KB 需要数十到数百小时的文献梳理。
- **KB 覆盖的组织有限。** Fuxi 目前只内置了视网膜 KB。如果你研究的是胰腺、大脑皮层或任何其他组织，KB 模式不可用。
- **KB 面对新的细胞状态无能为力。** 如果数据中出现了一个疾病特异性的细胞状态，KB 中没有任何条目对应它。

专家规则也有类似的问题：它们过于刚性，无法适应数据中出现的意外模式。

**AI 注释的价值就在于填补这个空白。** 一个预训练的 LLM（GPT-4o、DeepSeek 等）已经"读过"了数百万篇论文，它对大多数组织的细胞类型和标记基因有广泛的知识。即使在没有任何预先构建 KB 的情况下，它也能给出合理的注释初稿。

## Fuxi 的 AI 注释是如何工作的？

### 提示词（Prompt）结构

Fuxi 将提示词设计成三大部分：

**1. 系统提示词（System Prompt）**：设定角色和输出格式。例如 ATAC 版本的开头：

> "You are an expert in single-cell epigenomics and chromatin biology. Your task is to annotate ATAC-seq clusters based on their marker peaks and genomic features."

RNA 版本则设定为单细胞 RNA-seq 生物学家角色。

**2. 用户提示词（User Prompt）**：将数据中的信息填入模板：
- 组织名称（从配置中读取）
- 物种（从配置中读取）
- 每个聚类的 top marker 基因（RNA）或 top marker 峰区域（ATAC）
- 每个聚类的细胞数量

**3. 输出格式指令（Output Format）**：要求输出严格的 JSON 对象，包含以下字段：

```json
{
  "cell_type": "Photoreceptor",
  "state": "mature",
  "subtype": "rod",
  "confidence": "high",
  "reasoning": "High accessibility at RHO and NRL loci"
}
```

要求返回严格的 JSON 对象：

- `cell_type`：较宽的细胞类型名（如 "T cell", "Photoreceptor", "Müller Glia"）
- `state`：激活状态或功能状态（如 "resting", "activated", "proliferating", "mature", "N/A"）
- `subtype`：最具体的亚型（如 "CD8+ cytotoxic T cell", "rod photoreceptor", "N/A"）
- `confidence`："high", "medium", "low" 之一
- `reasoning`：一句话解释，引用关键的标记基因

```json
{"0":{"cell_type":"T cell","state":"activated","subtype":"CD8+ cytotoxic T cell","confidence":"high","reasoning":"High CD8A, GZMB, PRF1 expression indicates cytotoxic T cells"}}
```

### AI 缓存：为什么你的第二次运行不花钱

Fuxi 的 AI 系统包含一个磁盘缓存层。每次请求前，Fuxi 计算输入 prompt 的 SHA-256 哈希值，在缓存目录中查找是否有匹配的响应文件：

- **命中的情况**：直接加载缓存的 JSON 结果——0 秒延迟、0 费用
- **未命中的情况**：调用 LLM API，将响应写入缓存以备后续请求使用

这意味着如果你：
- 第一次运行 AI 注释后觉得聚类不够好，调整参数重新聚类，然后重新运行 Step 05——**只有新的聚类结果（不同的 marker 基因列表）会产生新的 API 调用**；相同的聚类会被缓存命中
- 同事在另一个项目中使用完全相同的配置和数据——缓存文件可以在团队中共享

### 模型自动发现

Fuxi 使用 OpenAI SDK 兼容的 API。系统会自动检测可用的模型列表（如果收到 404 错误会尝试自动发现），因此使用 OpenAI、DeepSeek、Together、vLLM、Ollama 等不同后端时无需手动配置模型名称——Fuxi 会检测可用模型并选择最合适的。

当启用思考模式（thinking mode）时，Fuxi 会自动将 `max_tokens` 提高到 32768，给模型更多空间进行推理。

## 什么时候应该信任 AI 注释？

AI 注释在以下场景中最为可靠：

- **LLM 识别的标记基因与已知生物学一致。** 如果 LLM 说"Cluster 2 是光感受器，因为 RHO、NRL 和 CRX 高表达"，而你确实在 marker 列表中看到了这些基因——这是高置信度的证据。
- **多个独立的标记基因指向同一个结论。** 一个基因可能是偶然的，但 5-10 个独立的 TFs/结构基因都指向同一个细胞类型，可信度很高。
- **置信度评分为 "high"。** LLM 自己给出的置信度评分总体上是有意义的（尽管不完美）——高置信度的预测通常值得优先采用。

## 什么时候不能信任 AI 注释？

### 1. LLM 幻觉

LLM 有时会编造听起来合理但实际上不存在的细胞类型。例如，它可能为视网膜中的一个聚类分配一个"视网膜自然杀伤细胞（retinal NK cell）"——这在生物学上可能完全不存在（没有文献支持），但基因标记听起来合理。这是大语言模型已知的"幻觉"问题。

**应对方式：** 最终注释应该由领域专家审核，特别关注那些"听起来很具体但不太常见"的细胞类型名称。

### 2. 小聚类的"猜测"

如果一个聚类只有几十个细胞，其中大部分基因标记是非特异的（核糖体蛋白、热休克蛋白等 housekeeping 基因），LLM 的判据很薄弱。这种时候 LLM 倾向于"猜测"——它在训练数据中学到了模式，但置信度极低。Fuxi 在 prompt 中包含了每个聚类的细胞数量，所以 LLM 会了解这是一个小聚类，但并非所有的 LLM 都可靠地处理这种信息。

### 3. 跨物种分析中的偏差

当前 LLM 的训练数据主要基于人类和小鼠。如果你分析的是斑马鱼、青蛙或果蝇的单细胞数据：

- LLM 会"默认"使用人类/小鼠的标记基因知识库（即使你告诉它物种不同）
- 基因符号可能不匹配（同源基因在非模式生物中有不同名称）
- 即使在 Fuxi 使用了同源基因映射（`ortholog.py`），那些被标记为 `UNMAPPED_` 前缀的基因在 prompt 中会显著降低 LLM 的注释信心

**应对方式：** 在跨物种分析中，AI 注释应被视为初步建议，需要手动验证。

### 4. 组织特异性变异

有些基因在不同组织中具有完全不同的功能。例如，PAX6 在大脑中是神经前体细胞的标记基因，在胰腺中是 beta 细胞的标记基因，在眼睛中是视网膜发育的关键调控因子。LLM 可能难以在 prompt 的上下文中准确区分这些组织特异性。

**应对方式：** 在配置中加入 `extra_context` 参数，提供额外的组织特异性背景信息。

## Fuxi 的安全保障机制

AI 注释虽然强大，但 Fuxi 设计上把它作为**替补方案**，不是主力。

```python
# 注释优先级（简略说明）
if tissue_kb 可用:
    # KB 模式：marker 打分 + 专家规则
    # AI 仅用于 KB 和规则都无法判定的聚类
elif AI 可用:
    # AI 模式：直接调用 LLM
else:
    # Score_genes 模式：纯标记基因打分
```

这种设计的核心理念是：

> **用确定性方法覆盖已知生物学的部分，用 AI 填补未知和边缘情况。**

KB + 规则引擎（确定性、可解释、可审计）优先处理 80-90% 的聚类——那些有明确标记基因的已知细胞类型。LLM 只被调用来处理剩余 10-20% 的"疑难"聚类（特征不明确、可能的新细胞类型等）。

这大幅降低了对 AI 输出的依赖风险：即使 AI 在某些聚类的注释上有误，主拓扑结构（已知细胞类型的正确标注）已经被 KB 保证了。

## Prompt 设计思路

### RNA 注释的默认键提示策略

RNA prompt 中，每个聚类的 top 20 marker 基因（Wilcoxon 秩和检验获得）构成了 LLM 看到的核心信息。Fuxi 有参数可选"compact mode"（仅前 5 个基因），但默认是 20 个，因为：

- 对于高置信度的聚类，前 5 个基因已经足够（但增加背景不会有害）
- 对于边缘聚类，20 个基因给 LLM 提供了更多线索
- 基因命名空间本身带有信息——例如前缀 "UNMAPPED_" 表示同源基因映射失败，LLM 知道降低自信度

### 约束命名（constrained naming）

当 KB 模式提供了候选细胞类型列表时，Fuxi 可以把这个列表注入到 AI prompt 中，指示 LLM **仅从该列表中选取名称**（constrained naming mode）。这保证了：
- AI 输出的名称与 KB 命名一致
- 不会发明全新的类别名称
- 如果一个聚类在列表中找不到匹配，LLM 输出 "Uncertain" 会被人工审核

> Fuxi 也支持非约束模式（unconstrained mode），此时 AI 可以建议列表外的细胞类型，但会加上 `[NOVEL]` 前缀。这在发现新细胞类型时很有用，但需要后续验证。

## 模型选择

Fuxi 的 AI 注释支持多种后端：

| 后端 | 推荐场景 | 注意事项 |
|------|---------|---------|
| OpenAI (GPT-4o/mini) | 标准使用，最稳定 | 需要 API Key，有使用费用 |
| DeepSeek-V3 | 高性价比替代 | 质量好，价格低 |
| vLLM | 自有 GPU 部署 | 需要本地硬件，延迟低但配置复杂 |
| Ollama | 完全本地运行 | 模型质量取决于选择，小型模型质量有限 |

**经验法则：** 对于生产环境的研究，推荐使用 GPT-4o 或等效的大型模型。本地 Ollama + 小型模型适合快速原型验证，但不适用于发表级的注释（除非经过严格的人工复核）。

## 常见误区

**"AI 注释结果可以不经审核直接使用。"** 绝对不能。AI 注释应该被视为**初稿**——一个有经验的研究人员应该审核每个聚类，确认标记基因与分配的细胞类型一致，修正 LLM 的明显错误。

**"AI 注释是纯粹的黑箱，完全不透明。"** 不是。Fuxi 的 prompt 中包含了推理所需的全部信息（top 标记基因、聚类大小），而 LLM 输出的 `reasoning` 字段解释了它做出判断的依据。你可以（也应该）看这些推理——如果推理说"因为 RHO 高表达"，你应该确认 RHO 确实在该聚类的 marker 列表中。

**"出现 LLM 错误说明这个系统没用。"** 错。与完全不使用 AI 相比（需手动查阅所有文献），即使有 80% 的准确率，AI 注释也能减少 80% 的审阅工作量。关键是后期的人工审核，而不是"完全信任"。

**"每次重新聚类都应该触发一次 API 调用。"** 不需要——Fuxi 的缓存系统确保相同输入的查询直接从磁盘加载，不会产生重复的费用和延时。

## 小结

- AI 注释将大类语言模型的生物学知识应用于细胞类型注释，特别适用于没有现成知识库的组织
- Fuxi 的 AI 系统基于精心设计的 prompt：系统角色 + 标记基因数据 + 严格 JSON 输出格式
- **缓存系统**通过 SHA-256 哈希避免重复 API 调用；**模型自动发现**简化了不同后端的配置
- 信任 AI 的前提：标记基因一致性高、多个独立证据、置信度高
- 需要警惕 AI 的情况：稀有聚类、跨物种、LLM 幻觉、组织特异性名称
- Fuxi 将 AI 作为 KB + 规则引擎的**替补**，而非主力——确定性方法优先处理已知生物学，仅将边缘案件交给 AI
- 约束命名模式（constrained naming）确保 AI 输出与知识库命名一致；非约束模式可用于发现新细胞类型
