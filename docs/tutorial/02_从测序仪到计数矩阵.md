# 从测序仪到计数矩阵

> 组织离体到你最终面前一张 Excel 表格——中间经过了哪些步骤？每一步丢失了什么信息？

## 我们要解决什么问题？

在上一章[中心法则与单细胞测序](01_中心法则与单细胞测序.md)中，我们说 Fuxi 的输入是"计数矩阵"（count matrix）——一个以基因为行、细胞为列的整数表格。但一个简单的疑问自然浮出水面：

**这个表格是怎么从一管组织变成数字的？**

如果你不了解这个过程的"黑箱"，你就很难理解你的数据为什么长这样——为什么有的细胞有 500 个基因被检测到、有的有 5,000 个？为什么同一种细胞类型在不同批次中测出的基因数差异很大？为什么 raw data 是整数，而有些论文中看到的是连续小数？

本章打开这个黑箱的第一层。我们不聊机器参数和算法细节（那是 Cell Ranger 的手册在做的事），但我们会走一遍**从组织样品到计数矩阵的关键步骤**，让你理解：每一个数字背后发生了什么，以及这个过程中哪些环节会引入噪声。

## 背后的原理

### 第一步：组织解离（tissue dissociation）——把细胞一个个分开

你的样品最初是一小块组织——视网膜、肝脏、肿瘤组织块。细胞之间由细胞外基质（extracellular matrix）黏合在一起。为了让测序仪能一个一个地处理细胞，你必须先把它们分开。

这通常通过**酶消化（enzymatic digestion）**实现：用胶原酶、胰蛋白酶等消化掉细胞间连接的蛋白质，然后机械吹打，得到**单细胞悬液（single-cell suspension）**。

> **丢失第一层信息：** 解离过程本身会改变细胞的转录状态。有些基因（如即时早期反应基因）在几分钟内就会因应激而上调。这不是你的"实验条件"造成的，是"把细胞从组织中摘出来"这个动作本身造成的。这是单细胞实验的固有困境——所有测量都会扰动被测量的对象。

### 第二步：细胞捕获与裂解——10X 液滴系统

目前最主流的技术是 10X Genomics 的 Chromium 平台。它的核心思路极其巧妙：**把单个细胞和单个"条形码珠"（barcode bead）一起包裹在一个油滴中。**

想象一个珍珠奶茶店——店家不是一杯一杯地做奶茶，而是把一杯分量的珍珠、冻粉和糖浆分别装进独立的小塑料袋，然后一次性把很多袋倒进热水中同时完成泡制。10X 也一样：它设计了一个微流控芯片（microfluidic chip），细胞悬液、酶试剂和条形码珠从三个通道流入，在交汇处形成油包水液滴（droplet）。理想情况下，**每个液滴恰好包裹一个细胞 + 一个条形码珠**。

条形码珠（barcode bead）是一颗覆盖了数十万条 DNA 引物的微珠，每条引物上都有一个**细胞条形码（cell barcode）**和**唯一分子标识符（UMI，Unique Molecular Identifier）**。所有来自同一个珠子的引物共享相同的条形码——相当于给这个细胞中的所有 mRNA 打上了同一个"邮政编码"。

液滴内，细胞被裂解，mRNA 释放出来，与珠子上的引物结合。逆转录（reverse transcription）发生——mRNA 被转化为更稳定的 cDNA（互补 DNA），同时带上了那条引物上的 barcode 和 UMI。

> **丢失第二层信息：** 不是每个液滴恰好有一个细胞。有些是空的（empty droplet——无细胞），有些是双细胞（doublet——两个细胞挤进一个液滴）。空液滴在后续分析中会被过滤掉（见[第二卷：质量控制](06_质量控制.md)），但双细胞需要专门的算法来检测（见[第二卷：双细胞检测](05_双细胞检测.md)）。

### 第三步：文库构建与扩增

液滴破裂后，所有带条形码的 cDNA 被收集到一起。接下来进行 PCR 扩增（polymerase chain reaction，聚合酶链式反应）——因为单个细胞中的 mRNA 量非常少（一个细胞大约有 10-50 万条 mRNA 分子），测序仪需要足够的 DNA 量才能读取。PCR 就是这个"复制"步骤。

每条 cDNA 片段上还需要加上测序接头（adaptor）——这是测序仪"读"DNA 所需要的"起读标记"。

> **丢失第三层信息：** PCR 扩增不是一个完美的复制过程。有些片段被复制得更多（PCR bias），有些很少。这就是 UMI（唯一分子标识符）存在的意义——UMI 是一段随机序列，每条原始 mRNA 分子在逆转录时都会带上一个不同的 UMI。在后续计数时，看到两条 cDNA 有相同的序列 + 相同的 UMI，它们被认为是同一条原始 mRNA 的复制品，只计 1 次。UMI 让计数更准确，但仍然不能完全消除偏倚。

### 第四步：测序（Illumina sequencing）

文库被加载到 Illumina 测序仪的流动槽（flow cell）上。测序的原理大致是：DNA 片段在芯片上被扩增成簇（cluster），然后通过"合成测序"（sequencing by synthesis）——每次加入一个荧光标记的碱基，机器拍照识别加上了什么碱基——逐碱基地读出 DNA 序列。

测序有三种读段（reads）：

- **Read 1**：读 cDNA 插入片段的一端（通常是基因序列）
- **Read 2**：读 cDNA 插入片段的另一端（如果是双端测序）
- **Index Read**：读条形码和 UMI

测序的直接输出是 **FASTQ 文件**——一个巨大的文本文件，里面每四条记录是一个 DNA 片段的完整序列和对应的质量分数。

> 从 FASTQ 到计数矩阵，这一步不是 Fuxi 做的。Fuxi 在"之后"介入。

### 第五步：从 FASTQ 到计数矩阵——Cell Ranger / STARsolo 的工作

FASTQ 文件中的序列除了包含基因的序列，还包含：

- **细胞条形码（cell barcode）**：告诉你在哪个细胞中测到的
- **UMI**：告诉你这是一条原始 mRNA 还是它的 PCR 副本
- **基因序列**：告诉你这来自哪个基因

**比对（alignment）** 工具（如 STAR、Cell Ranger）将基因序列与参考基因组（ref genome）进行匹配，确定它来自哪条基因。因为是 RNA 测序，比对还需要考虑**剪接（splicing）**——成熟的 mRNA 已经去除了内含子（intron），所以比对时允许 reads 跨越外显子-外显子连接（exon-exon junction）。

然后，**去重（deduplication）**：同一细胞、同一基因、同一 UMI 的多条 reads 只计 1 次。这一步去除了 PCR 扩增带来的重复计数。

最终，所有经过上述步骤的信息被汇总为每个细胞中每个基因的**唯一 UMI 计数**。这就是你的计数矩阵。

## Fuxi 是怎么做的？

Fuxi 不做以上 1-5 步。Fuxi 从步骤 5 的**输出**开始。具体来说，Fuxi 接受以下输入格式：

| 格式 | 典型文件 | 来源 |
|------|----------|------|
| `10X_h5` | `filtered_feature_bc_matrix.h5` | Cell Ranger 标准输出，一个 HDF5 文件 |
| `10X_mtx` | `matrix.mtx.gz` + `barcodes.tsv.gz` + `features.tsv.gz` | Cell Ranger 原始输出，三个文件 |
| `csv_matrix` | 基因 × 细胞计数矩阵 CSV/TSV | 非 10X 平台，如 Smart-seq2 |
| `h5ad` | `*.h5ad` | AnnData 格式——已被其他工具处理过的数据 |

关于**稀疏矩阵（sparse matrix）**，一个直观的理解：想象一个有 20,000 列（基因）× 10,000 行（细胞）的 Excel 表格，但 90% 以上的格子是 0。如果用普通矩阵存储，你需要 20,000 × 10,000 × 8 字节 ≈ 1.6 GB。但稀疏矩阵只存非零值以及它们的行列位置，同样数据可能只占几十 MB。

Fuxi 使用 AnnData 的稀疏矩阵格式（`scipy.sparse.csr_matrix`）存储计数矩阵。这是一个标准化的实践，但你需要理解它的含义：你仍然在做"所有基因"的下游分析——高变基因选择、PCA、聚类——只是计算上高效得多。稀疏格式不影响生物学解释，它只是一个存储和计算的策略。

> 具体参数配置见 [Pipeline 指南](../pipeline_guide_zh-CN.md#step-00数据加载)

### 不支持 R 格式（.rds / .qs）

这是一个常见的痛点。很多公共数据库中 scRNA-seq 数据以 R 的 `.rds` 或 `.qs` 格式存储（Seurat 对象的 Serialization 格式）。Fuxi 不原生支持它们，原因很简单：Fuxi 是基于 Python 生态构建的（AnnData / Scanpy），而 `.rds` 格式封装的是 R 内部的数据结构，无法直接被 Python 解析。

解决方案：使用 [r2h5ad](https://github.com/nanshanteahouse/r2h5ad) 转换工具——在 R 环境中读取 `.rds`，输出为 Fuxi 兼容的 `.h5ad` 文件。

## 我怎么看结果？

加载完成后，Fuxi 会打印出数据的基本信息：

```
AnnData object with n_obs × n_vars = 10000 × 20000
    var: 'gene_symbols'
    obs: 'sample', 'stage'
```

- `n_obs` 是细胞数——但这包括了空液滴和双细胞，需要在后续步骤中过滤
- `n_vars` 是基因数——通常是 20,000-30,000 个蛋白质编码基因
- `obs` 中的信息（如 `sample`、`stage`）来自 `CFG.sample_map` 和 `CFG.stage_map` 的配置

在 `results/h5ad/00_raw.h5ad` 文件中，你可以随时重新加载这个数据（通过 Scanpy 的 `sc.read()` 函数）来查看原始矩阵。

## 常见误区

**"计数 = 表达量"**——计数是 UMI 去重后的 mRNA 分子计数，不是表达水平。同一个细胞中的不同基因有不同的检测效率（取决于捕获效率、GC 含量、转录本长度等），不同细胞之间的计数也不可直接比较（因为细胞大小不同，总 mRNA 量不同）。这就是为什么需要后续的**归一化**步骤。

**"FASTQ → 计数矩阵 = 自动化，没有人为干预空间"**——决定在比对时使用什么参考基因组版本（hg38 vs mm10 vs 自定义注释）、是否包含 pre-mRNA（内含子 reads）、UMI 去重的阈值——这些都会影响最终的计数矩阵。这也是为什么不同工具（Cell Ranger vs STARsolo vs Kallisto）输同一个数据时计数不完全相同。

**".h5 文件就是 mtx 三个文件的打包"**——不完全是。`filtered_feature_bc_matrix.h5` 确实包含了与三个 `.tsv/.mtx` 文件相同的信息（barcodes × features × matrix），但 `.h5` 格式（HDF5）本身是一个通用的层级化数据存储格式，可以包含更复杂的结构。Fuxi 对两种格式都支持，并且会统一转化为 `.h5ad`（AnnData 的标准存储格式），以保持内部数据管道的一致性。

## 小结

- 从组织样品到计数矩阵经历了：**解离 → 液滴捕获 → 逆转录 → 文库扩增 → 测序 → 比对 → UMI 去重**
- 每一步都会引入特定的噪声和偏倚：解离应激、双细胞、PCR bias、基因检测效率差异
- Fuxi 从处理后的计数矩阵开始工作，接受 10X_h5、10X_mtx、csv_matrix、h5ad 四种输入格式
- 计数矩阵以稀疏矩阵存储以节省内存，但所有后续分析"假装"我们拥有完整的基因信息
- R 格式（.rds/.qs）需要通过 [r2h5ad](https://github.com/nanshanteahouse/r2h5ad) 转换后才能被 Fuxi 处理
