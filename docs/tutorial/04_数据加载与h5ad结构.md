# 数据加载与 h5ad 结构

> 第 00 步：把散落在磁盘各处的原始数据整理成一个统一、高效、可查询的容器。

## 我们要解决什么问题？

你从 GEO 或自己的测序平台拿到原始数据，通常不是单个文件。它可能是一堆文件：

- 一个矩阵（表达值表）
- 两三个元数据文件（barcode 列表、基因名列表）
- 对于 10X Genomics 的数据，是一个 HDF5 格式的 `.h5` 文件
- 如果手上有多个样本，每个样本又各自有一套这样的文件

更麻烦的是，不同的实验室可能用不同的格式给你数据。有的给 Cell Ranger 输出的 MTX 目录，有的给 CSV，有的直接给一个已经在别的软件里注释过的 `.h5ad` 文件。

Step 00 的任务，就是把这一切混乱的输入**统一成一个标准化的 AnnData 容器**，输出为 `.h5ad` 文件。从此以后，管线的后续所有步骤只跟这一个文件打交道。

## 什么是 AnnData？——一个丝滑的"细胞×基因"笔记本

AnnData 是专门为单细胞组学数据设计的 Python 数据结构。它的设计哲学很简单：**把关于一个数据集的所有信息放在同一个盒子里，分门别类，便于查找**。

你可以把它想象成一本非常规范的实验记录本：

| AnnData 组件 | 相当于笔记本的... | 存了什么？ |
|:--|:--|:--|
| `X` | 核心数据表 | 表达矩阵：行是细胞，列是基因，值是表达量 |
| `obs` | 细胞索引卡 | 每个细胞的属性：来自哪个样本、属于哪个批次、QC 指标等 |
| `var` | 基因索引卡 | 每个基因的属性：基因名、是否是高变基因、是否是线粒体基因等 |
| `obsm` | 降维草图本 | 降维结果：PCA 坐标、UMAP 坐标、Harmony 校正后的坐标 |
| `uns` | 杂项抽屉 | 不受上述分类约束的数据：聚类参数、颜色映射、基因集评分结果 |

### 为什么需要这种结构？

管线中每一步都可能产生新的信息。QC 步骤需要在细胞上加标注（"这个细胞是双细胞"），聚类步骤需要加聚类编号，注释步骤需要加细胞类型。如果每个步骤都操作不同的文件，很容易搞混。

AnnData 把这些信息**缝在一起**——`obs` 列的增长就是管线进展的日志。你随时可以按细胞类型分组、按样本分组、按 QC 状态过滤，而数据始终保持对齐。

## 稀疏矩阵：为什么能省这么多内存？

单细胞表达矩阵有个有趣的性质：**90%~99% 的条目是 0**。一个基因只在特定细胞类型中表达，一个细胞只表达几千个基因——所以大部分"细胞×基因"组合都等于零。

如果存储为一个普通的 20,000×20,000 的"密集"矩阵（double 精度），需要约 3.2 GB 内存（200M 个值 × 8 字节）。而用**稀疏矩阵**只存储非零元素的位置和值，通常只需要 ~100–200 MB。

> 一个理解的类比：想象你们实验室有 100 人。每个人手里有 20,000 张名片，但大多数人只有几百张是有字的。你没必要复印 100×20,000 张纸——只记录"谁有谁的名片"就够用了。

稀疏矩阵有两种常见存储格式：
- **CSR（Compressed Sparse Row，行优先）**：按行压缩，取一整行的数据很快（即查一个细胞的所有基因表达）
- **CSC（Compressed Sparse Column，列优先）**：按列压缩，取一整列的数据很快（即查一个基因在所有细胞的表达）

Scanpy 默认使用 CSR。Fuxi 在加载时也会统一转换成 CSR（`force_csr=True`），因为按细胞操作是单细胞分析最常见的工作模式。

## 什么是降采样（downsampling）？什么时候需要？

降采样就是从数据集中**随机抽取一部分细胞**。听起来像是在丢弃信息，但有时是必要的策略选择：

1. **内存限制**：100 万个细胞 × 2 万个基因的稀疏矩阵仍然需要几十 GB 内存。如果你的机器只有 32 GB，不得不降采样。
2. **探索阶段**：在调整参数阶段，先用 10% 的细胞快速跑一遍管线找到合适的参数组合，再全量跑完整流程。
3. **严重不平衡的数据集**：一个样本有 10 万细胞，另一个只有 1000。降采样可以避免大样本主导聚类结果。

Fuxi 支持三种降采样方式，在配置文件中控制：
- **随机采样（random）**：完全随机，适合单样本或不需要保持比例的场景
- **分层采样（stratified）**：按样本分组，保持各样本在降采样后比例不变
- **每个样本最多 N 个（max_per_sample）**：每个样本截断到上限，适用于严重不平衡

> **重要：** 降采样只在 Step 00 执行一次。之后的管线步骤操作的是降采样后的数据。如果你想全量分析，在配置中注释掉降采样参数即可。

## Fuxi 是怎么做数据加载的？

Step 00 支持四种输入格式，自动识别 + 差异化加载：

### 1. 10X MTX（Cell Ranger 标准输出）

最常见。目录里包含三个文件：`matrix.mtx.gz`、`barcodes.tsv.gz`、`features.tsv.gz`（旧版本是 `genes.tsv.gz`）。Fuxi 调用 `sc.read_10x_mtx()` 读取。

如果发现旧版 2 列的 `genes.tsv.gz`，它会自动转换成 3 列的 `features.tsv.gz`（多了一列 "Gene Expression"），保证兼容性。

⚠️ **关键步骤：barcode 后缀解析。** 10X 数据中，每个 barcode 末尾带有 `-1`、`-2` 这样的后缀，表示它来自不同的样本。Fuxi 会从 barcode 中提取这个数字，然后根据配置文件中的 `sample_map` 或 `stage_map`，把 `-1` 映射为 "Sample_A"、`-2` 映射为 "Sample_B"，存入 `adata.obs['sample']` 或 `adata.obs['stage']`。

```python
# 配置中你只需要定义：
CFG.sample_map = {1: "WT_D0", 2: "WT_D7", 3: "KO_D0", 4: "KO_D7"}
# Fuxi 自动完成 barcode → sample 映射
```

没有这个映射，后续的批次校正（Harmony）就不知道哪些细胞属于同一个样本。

### 2. 10X HDF5（`10X_h5` 格式）

Cell Ranger 也输出 `.h5` 文件（HDF5 格式），尤其在多芯片研究中常见。Fuxi 支持读取单个或多个 `.h5` 文件：

- **单个文件**：直接读取，从文件名提取样本名
- **多个文件**：逐一读取，加 `sample` 标注，然后 `sc.concat()` 合并

合并时会自动处理重复的基因名和 barcode（通过 `index_unique='-'` 参数给不同样本的细胞加上 `-1`、`-2` 后缀避免冲突）。

### 3. CSV 矩阵（`csv_matrix`）

如果你的数据是基因×细胞格式的 CSV 文件（行名是基因名，列名是 barcode），Fuxi 会读取后转置成细胞×基因的 AnnData 标准格式。

还可以额外提供 `barcodes_file` 和 `features_file` 附带元数据。

### 4. 已有 h5ad（`h5ad` 格式）

直接读取现成的 `.h5ad` 文件。适用于从上游流程拿到的已标注数据，或者断点续跑。

### 多格式处理的统一哲学

无论哪种格式，加载完成后 Fuxi 执行统一的收尾工作：
1. 统一转换为 CSR 稀疏矩阵
2. 可选转换为 float32 精度
3. 可选降采样
4. 确保 `obs_names` 唯一
5. 写入 `00_raw.h5ad` 作为管线第一个 checkpoint

**这样，Step 01 及之后的所有步骤都不需要关心原始数据长什么样。**

## 我怎么看结果？

Step 00 的主要产出就是 `00_raw.h5ad`。你可以通过日志确认这几个关键信息：

```
Step 00: Load raw data
Format: 10X_mtx
Loading complete: 12734 cells × 36601 genes
Sample mapping applied:
  WT_D0       4758 cells
  WT_D7       3921 cells
  KO_D0       2174 cells
  KO_D7       1881 cells
```

关注以下几点：
- **细胞数和基因数**：是否符合预期？如果细胞数明显少于预期的细胞回收量，检查数据文件是否完整
- **样本映射是否正确**：各个样本的细胞数分布是否合理？
- **降采样是否触发**：如果配置了降采样，日志会显示 "Random sampling: 12734 → 5000 cells" 之类的信息

## 常见误区

**误区 1："我只有一个样本的文件，不需要 barcode 后缀解析。"**
即使只有一个样本，10X 仍然会输出 `-1` 后缀。如果你不设置 `sample_map`，Fuxi 不会报错，但样本列会缺失——这不会影响 Step 00，但当你在 Step 03 尝试做批次校正时，会发现没有可用的批次分组列。

**误区 2："h5ad 文件很小，因为我的数据只有几百 MB。"**
`.h5ad` 文件在磁盘上可能只有几百 MB，但读入内存后展开成稀疏矩阵可能占用数 GB。这是因为 HDF5 压缩格式在磁盘上很紧凑，但读取时会解压。降采样在内存不足时非常有用。

**误区 3："我的 CSV 文件没有报错，加载成功了。"**
CSV 文件加载时不检查数据类型。如果 CSV 中的表达值是 log-normalized 而你不小心标记成了 raw counts，后续的 Scrublet 双细胞检测会出现误判。详见[双细胞检测](05_双细胞检测.md)章节。

**误区 4："加载完的 h5ad 和原始文件一样大，磁盘空间不够用。"**
`.h5ad` 文件的大小很大程度上取决于是否做降采样以及矩阵的稀疏程度。如果磁盘空间紧张，可以在配置中启用降采样，或者在保存时使用 `compression='gzip'` 参数。

## 小结

Step 00 是整个管线的基础——它把各种格式、各种分布的原始数据统一成一个标准的 AnnData 容器（`.h5ad`）。这个过程不仅是简单的格式转换，还包括 barcode→样本映射、降采样策略、稀疏格式统一等关键操作。

理解 AnnData 的结构（`X`、`obs`、`var`、`obsm`、`uns`）对你后续分析和调试管线都非常有帮助——几乎每一步操作都在为 AnnData 的某个组件增加新内容。
