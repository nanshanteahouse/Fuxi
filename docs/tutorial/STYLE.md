# Fuxi Wiki 写作风格指南

## 核心原则

- **回答"为什么"，不重复操作手册。** 关于"输入什么命令、配什么参数"的内容，已经在 `docs/pipeline_guide_zh-CN.md` 中。我们的 wiki 在必要时引用它，但不重复。
- **假设读者知道：** 什么是细胞、什么是 DNA/RNA/蛋白质、什么是基因。不需要解释这些。
- **不假设读者知道：** 什么是单细胞测序、什么是 UMI、什么是 batch effect、什么是 p-value。这些是我们要教的。
- **定义知识的边界：** 讲 UMAP 时不讲拓扑学，讲 PCA 时不写线性代数，讲 p-value 时不说"在原假设为真的情况下观察到极端结果的概率"。用直觉和类比。
- **所有专业术语第一次出现时（中文括号标英文），如"双细胞（doublet）"。**
- **目录风格：** 使用提问式或场景式标题，如"为什么 10X 数据一定有 doublet？"而非"1.1 Doublet 概述"。

## 章节结构模板

每章标准结构（可根据需要调整顺序或合并）：

```markdown
# 章节标题

> 一句话摘要——读者读完后应该理解的核心概念。

## 我们要解决什么问题？
（生物学背景和动机。这一节回答：为什么这个步骤存在？不做会怎样？）

## 背后的原理
（核心概念解释。用类比和直觉，不用数学公式。如果一定要涉及统计概念，用一句话说清直觉。）

## Fuxi 是怎么做的？
（Fuxi 的实现选择。回答：为什么不选其他方法？设计时考虑了哪些 trade-off？）
> 具体参数和命令见 [Pipeline 使用指南](../pipeline_guide_zh-CN.md#对应章节)

## 我怎么看结果？
（输出文件是什么、图怎么看、重点关注什么。直接告诉读者："看这张图，先看这里，再看那里。如果出现 X，意味着 Y。"）

## 常见误区
（读者最容易误解的地方）

## 小结
（3-5 句话收束。读者应该带走的核心概念。）
```

## 语气与风格

- **友好但有深度。** 像有经验的师兄/师姐在跟你解释，不居高临下也不敷衍。
- **慎用比喻，但要精准。** 一个类比用到底比三个类比轮着用好。例如"液滴捕获类似捞乒乓球——你希望一个捞一个，但有时两个球同时掉进同一个网兜"就比"泊松过程的概率密度函数"好一万倍。
- **承认不确定性。** "这部分领域还没有定论""Fuxi 在这里做了一个简化假设""这个参数的选择有一定主观性"。不假装所有决策都有完美答案。
- **长度：** 每章 1500-3000 字（中文）。太短说不透，太长没人读。

## 格式规范

- 文件编码：UTF-8
- 文件名：两位数字 + 英文短名，如 `01_中心法则与单细胞测序.md`
- 图片占位：`![图 X：描述文字](images/placeholder.png)`
- 引用操作手册：`> 具体参数配置见 [Pipeline 指南](../pipeline_guide_zh-CN.md#step-00数据加载)`
- 跨章节引用：使用 `[中心法则](01_中心法则与单细胞测序.md)`
- 强调用 `**`，不用 `*` 或 `__`
- 代码块用 ` ``` `

## 什么该写 / 什么不该写

| 该写（举例） | 不该写 |
|------|--------|
| 为什么 10X 的液滴捕获必然产生 doublet | Scrublet 的具体算法参数 |
| 为什么线粒体比例能指示细胞质量 | python 代码片段 |
| 为什么 Harmony 校正 batch effect 后数据变了（不只是"去掉了批次"） | pip install 命令 |
| 为什么 Leiden 聚类优于 K-means | config.py 中每一行的作用 |
| 什么是"伪时间（pseudotime）"的直觉 | 如何安装轨迹分析依赖 |
| 什么是 TF 活性 vs TF 表达量 | 如何配置 GRN 参数 |

底线：读者看完一章后，应该能在脑中画出一张概念地图，知道这个步骤**为什么存在、在解決什么问题、结果应该怎么看**。而不是知道这个步骤用什么命令跑。
